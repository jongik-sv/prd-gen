# Part 6: PRD 문서 작성 토론 종합 요약

## 개요
- **주제**: 비전/목표, 페르소나, 기능 요구사항(포털/권한/마스터), 비기능 요구사항, MoSCoW 우선순위, 릴리즈 로드맵, 리스크 관리
- **참여 전문가**: Andrew Ng, Andrej Karpathy, Martin Kleppmann, Martin Fowler, Robert C. Martin, Kent Beck, Jakob Nielsen, Bruce Schneier, Donald Knuth

---

## 1. 비전 및 목표

### 비전 스테이트먼트
"모든 구성원이 하나의 포털에서 모든 업무 시스템에 안전하게 접근하고, 일관된 권한과 신뢰할 수 있는 마스터 데이터로 효율적으로 협업하는 통합 업무 플랫폼"

### 비즈니스 목표 (B1~B4)
- B1: SSO 단일 로그인, 업무 진입 시간 75% 단축
- B2: 중앙 RBAC, 프로젝트별 권한 사일로 해소
- B3: 마스터 데이터 SSOT 확보
- B4: SDK 표준화, 온보딩 2주 이내

### 기술 목표 (T1~T6)
- T1: 가용성 99.9%, p95 < 200ms / T2: 독립 배포 / T3: Zero Trust / T4: Contract-First / T5: Tracer Bullet / T6: OpenTelemetry

---

## 2. 페르소나 (4종)

- **김서연** (최종 사용자): 빠른 시스템 전환, 통합 알림
- **박정우** (프로젝트 관리자): 권한 관리, 퇴직자 자동 회수(JML)
- **이승호** (시스템 관리자): 전사 모니터링, 온보딩 2주 단축
- **최민지** (개발자): 명확한 API 문서, TypeScript 타입, 하위 호환성

---

## 3. 기능 요구사항 (26개 User Story)

### 포털 (US-P01~P09, 9개)
- 대시보드, 런처/앱전환, 통합검색(Cmd+K), 글로벌 알림, 프로필/설정, 최근접근

### 권한관리 (US-A01~A09, 9개)
- SSO 로그인, MFA, 역할 관리, 역할 배정, 내 권한 조회, 권한 위임(최대 30일, 깊이 1), 요청/승인, 감사 로그

### 마스터관리 (US-M01~M08, 8개)
- 코드 CRUD, 이력 관리, 벌크 처리(5,000건), 승인 워크플로우, 데이터 품질 대시보드

---

## 4. 비기능 요구사항

| 항목 | 수치 |
|------|------|
| API 응답 | p50 < 100ms, p95 < 200ms, p99 < 500ms |
| 동시 접속 | 일반 1,000명, 피크 5,000명 |
| Core Web Vitals | LCP < 2.5s, FID < 100ms, CLS < 0.1 |
| SLA | 포털 99.9%, 인증 99.95% |
| RTO/RPO | < 30분 / < 5분 |
| 보안 | OWASP Top 10, TLS 1.3, AES-256, Zero Trust |
| 접근성 | WCAG 2.1 AA |

---

## 5. MoSCoW 우선순위

- **Must Have** (12개 + 기술 기반): SSO, 기본 RBAC, 대시보드, 코드 CRUD, Shell+MFE, 캐시, 감사 로그, SUPER_ADMIN MFA, OpenTelemetry
- **Should Have** (9개): MFA 확대, 위임, 워크플로우, 검색, 알림, 이력, 감사 상세, 설정
- **Could Have** (7개): 벌크, 품질 대시보드, AI 검색, 권한 추천, 위젯, JML
- **Won't Have** (6개): 크로스 프로젝트, BI, 모바일, 마켓플레이스, 멀티 테넌트, 실시간 협업

---

## 6. 릴리즈 로드맵 (5단계, 총 42주)

| Phase | 기간 | 핵심 내용 |
|-------|------|----------|
| **0** | 4주 | Tracer Bullet (SSO -> Shell -> Remote App -> 마스터코드) |
| **1** | 8주 | SSO 완성, Shell App, SDK v0.1, 기본 RBAC, 코드 CRUD, 파일럿 1개 |
| **2** | 8주 | 커스텀 역할, 워크플로우, 검색, 알림, 벌크, SDK v1.0 |
| **3** | 10주 | 3~5개 프로젝트 온보딩, 권한 위임, JML, 품질 대시보드 |
| **4** | 12주 | AI 검색, AI 권한 추천, 데이터 품질 자동화, ABAC |

- 각 Phase에 Go/No-Go 기준 명확히 정의

---

## 7. 리스크 관리 (8개)

| 리스크 | 영향도 | 대응 |
|--------|--------|------|
| IdP 연동 지연 | 높음 | Mock IdP 병렬 개발 |
| Module Federation 호환성 | 높음 | Phase 0에서 조기 검증 |
| 온보딩 저항 | 중간 | 파일럿 성공 사례 + SDK DX |
| 데이터 마이그레이션 | 높음 | 점진적, 양방향 동기화 |
| 성능 미달 | 중간 | 자동화 테스트, Phase별 벤치마크 |
| AI 정확도 | 낮음 | Phase 0부터 데이터 수집 |
| 보안 취약점 | 높음 | 정기 감사, 자동 스캔 |
| 핵심 인력 이탈 | 높음 | Clean Code, 페어 프로그래밍 |

## 핵심 원칙
- **AI 원칙**: AI는 추천까지만, 보안 결정은 반드시 사람이 내린다
- **연쇄 위임 금지**: 깊이 1단계
- **Critical 보안 알림**: 사용자가 비활성화 불가
