## 개요

순수 탭(TDI: Tabbed Document Interface) 방식은 화면 상단에 탭 바가 있고, 한 번에 하나의 탭 내용만 콘텐츠 영역에 표시되는 레이아웃이다. 웹 브라우저의 탭 UX와 동일한 패러다임으로, 대부분의 사용자에게 직관적이다.

## 레이아웃 구조

```
┌──────────────────────────────────────────────┐
│  Header  [통합검색] [알림] [프로필]            │
├──────────────────────────────────────────────┤
│  Tab Bar                                      │
│  [생산현황 ●] [작업지시] [품질검사 ◉] [+]       │
├─────┬────────────────────────────────────────┤
│     │  Content Area                          │
│  S  │  ┌────────────────────────────────┐    │
│  i  │  │                                │    │
│  d  │  │   활성 탭의 Remote App 표시      │    │
│  e  │  │                                │    │
│  b  │  │   (나머지 탭은 display: none     │    │
│  a  │  │    으로 숨겨져 상태 유지)         │    │
│  r  │  │                                │    │
│     │  └────────────────────────────────┘    │
└─────┴────────────────────────────────────────┘
```

## 탭 시스템 동작

### 탭 생성

- 사이드바 메뉴 클릭 시 새 탭으로 열림 (기존 탭 유지)
- 같은 메뉴를 다시 클릭하면 기존 탭을 활성화 (중복 방지)
- Ctrl+Click으로 같은 화면의 새 인스턴스를 강제 생성 가능
- 최대 탭 수 제한: 기본 10개 (설정 가능), 초과 시 가장 오래된 비활성 탭 닫기를 제안

### 탭 전환

- 클릭으로 탭 전환
- Ctrl+1~9: 탭 번호로 직접 이동
- Ctrl+Tab / Ctrl+Shift+Tab: 다음/이전 탭
- 전환 시 이전 탭은 unmount하지 않고 숨기기만 하여 상태 보존

### 탭 상태 보존

탭 전환 시 다음 상태가 모두 유지된다:
- 폼 입력 중인 데이터
- 스크롤 위치
- 테이블 정렬/필터 상태
- 로딩된 데이터 캐시

### 탭 표시 정보

| 요소 | 설명 |
|------|------|
| 아이콘 + 화면명 | 기본 제목 |
| ● 표시 | 수정 중인 미저장 데이터가 있음 |
| ◉ 표시 | 실시간 데이터 수신 중 (라이브 상태) |
| 숫자 배지 | 알림/미처리 건수 |

### 탭 관리

- 탭 드래그로 순서 변경
- 우클릭 컨텍스트 메뉴: 닫기, 다른 탭 모두 닫기, 새 창에서 열기(Pop-out)
- 닫기 시 미저장 데이터가 있으면 확인 대화상자 표시

## 메모리 관리

탭 수가 많아지면 메모리가 부족해질 수 있으므로 LRU(Least Recently Used) 기반 3단계 관리를 적용한다:

| 단계 | 상태 | 설명 |
|------|------|------|
| **활성(Active)** | DOM 표시 | 현재 보고 있는 탭 |
| **숨김(Hidden)** | DOM 유지, 숨김 | 최근 사용한 탭, 전환 시 즉시 복원 |
| **휴면(Suspended)** | 상태 직렬화 후 unmount | 오래 사용하지 않은 탭, 재활성화 시 재마운트 |

최대 활성 탭 수는 메모리 사용량에 따라 동적으로 조절되며, 힙 사용량이 임계값(예: 1.5GB)을 초과하면 LRU 순서로 자동 휴면 처리한다.

## Pop-out (새 창에서 열기)

탭을 별도 브라우저 창으로 분리하는 기능으로, 듀얼 모니터 환경에서 유용하다. Pop-out 창은 경량 Shell(최소 헤더 + 단일 앱)로 구성되며, 메인 Shell과 BroadcastChannel로 인증 상태와 이벤트를 동기화한다.

## 제약 사항 및 고려사항

1. **동시 참조 불가**: 두 화면을 나란히 비교해야 하는 사용 시나리오를 지원할 수 없음 (Pop-out으로 부분 해결 가능)
2. **산업 현장 부적합**: MES 운영자 등 동시 다화면 작업이 필수인 사용자에게는 기능 부족
3. **마스터-디테일**: 목록에서 항목 선택 → 상세 보기를 같은 화면에서 하려면 해당 앱이 자체적으로 분할 레이아웃을 구현해야 함
4. **Pop-out 의존**: 동시 보기가 필요하면 Pop-out에 의존해야 하는데, 창 관리가 번거로울 수 있음
