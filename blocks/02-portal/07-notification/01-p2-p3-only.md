  - 산업 현장의 안전 관련 규정을 충족하지 못할 수 있음

  - 알림의 심각도가 대부분 낮은 시스템 (승인 요청, 작업 완료 등)
  - 초기 MVP에서 기본 알림만 먼저 구현하고 긴급 알림은 추후 추가하려는 경우

## 개요

일반 알림(P2-P3) 전략은 알림 센터에 알림을 쌓아두고 사용자가 확인하는 풀(Pull) 방식을 기본으로 한다. 헤더의 알림 아이콘에 미확인 건수 배지를 표시하여 알림이 있음을 알려주고, 사용자가 아이콘을 클릭하면 알림 센터 패널이 열린다.

## 알림 수준별 동작

### P2 (Normal)

| 항목 | 동작 |
|------|------|
| 시각적 피드백 | 헤더 알림 아이콘에 숫자 배지 표시 |
| 알림 센터 | 새 알림이 목록 상단에 추가 |
| 토스트 | 옵션: 화면 우상단에 자동 사라지는 토스트 표시 (3~5초) |
| 사운드 | 없음 |
| 액션 | 알림 클릭 시 관련 화면으로 이동 (예: 승인 요청 → 승인 화면) |

### P3 (Low)

| 항목 | 동작 |
|------|------|
| 시각적 피드백 | 알림 센터에만 쌓임 (배지 숫자에 포함되지 않을 수 있음) |
| 알림 센터 | 새 알림이 목록에 추가 |
| 토스트 | 없음 |
| 사운드 | 없음 |
| 액션 | 알림 클릭 시 상세 보기 또는 관련 링크로 이동 |

## 알림 센터 구성

### 패널 구조

```
┌─────────────────────────┐
│  알림 센터               │
├─────────────────────────┤
│  [전체] [미확인] [유형별] │
├─────────────────────────┤
│  ● 작업지시 #1234 완료    │
│    2분 전 | 생산관리      │
├─────────────────────────┤
│  ● 승인 요청 - 자재 발주  │
│    15분 전 | 구매팀       │
├─────────────────────────┤
│  ○ 시스템 업데이트 안내    │
│    1시간 전 | 시스템       │
├─────────────────────────┤
│  [모두 읽음으로 표시]      │
│  [알림 설정]              │
└─────────────────────────┘
```

### 알림 항목 정보

| 필드 | 설명 |
|------|------|
| 제목 | 알림의 한 줄 요약 |
| 설명 | 상세 내용 (옵션) |
| 시각 | 발생 시각 (상대 시간: "2분 전") |
| 출처 | 발생 앱 또는 모듈 |
| 읽음 상태 | ● 미확인 / ○ 확인 |
| 액션 URL | 클릭 시 이동할 경로 |

### 알림 관리

- 읽음/미읽음 토글
- 개별 삭제, 전체 읽음 처리
- 유형별 필터 (작업, 승인, 시스템 등)
- 알림 설정: 유형별 수신 ON/OFF, 토스트 표시 여부

## 알림 전달 방식

| 방식 | 실시간성 | 구현 복잡도 | 적합 |
|------|---------|-----------|------|
| **폴링 (Polling)** | 주기적 (30초~1분) | 낮음 | P3 수준 |
| **SSE (Server-Sent Events)** | 준실시간 | 중간 | P2-P3 수준 |
| **WebSocket** | 실시간 | 높음 | P0-P3 전체 |

P2-P3만 지원한다면 SSE 또는 폴링으로 충분하다. WebSocket은 P0-P1 지원 시 필요하다.

## 제약 사항 및 고려사항

1. **긴급 대응 불가**: 설비 이상, 품질 이탈 등 즉각 대응이 필요한 상황에 대한 알림 수단이 없음
2. **사용자 인지**: 토스트가 없거나 짧으면 알림을 놓칠 수 있음
3. **알림 과다**: 필터링 없이 모든 알림이 쌓이면 중요 알림이 묻힐 수 있음
4. **오프라인**: 브라우저를 닫은 상태에서의 알림은 다음 접속 시에만 확인 가능
