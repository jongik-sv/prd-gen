## 개요

전 디바이스 지원 정책은 320px(소형 스마트폰)부터 4K 모니터까지 모든 뷰포트를 공식 지원하는 전략이다. Mobile-First 또는 Desktop-First 접근 방식으로 설계하되, 모든 브레이크포인트에서 기능이 사용 가능해야 한다.

이 전략의 핵심 과제는 **"기능 적응(Feature Adaptation)"**이다. 데스크톱의 모든 기능을 모바일에 그대로 넣는 것이 아니라, 디바이스별로 사용 가능한 기능 범위를 차등 정의한다.

## 디바이스별 기능 적응

| 기능 | 데스크톱 (1280px+) | 태블릿 (768~1279px) | 모바일 (~767px) |
|------|-------------------|-------------------|-----------------|
| 사이드바 | 펼침 | 접힘 (햄버거) | 하단 탭 바 |
| Split View | 최대 4분할 | 최대 2분할 | 미지원 |
| 탭 바 | 상단 탭 바 | 상단 탭 바 | 스와이프 전환 |
| Pop-out | 새 창 | 미지원 | 미지원 |
| 키보드 단축키 | 전체 | 부분 | 미지원 |
| 데이터 테이블 | 전체 컬럼 | 주요 컬럼 | 카드 뷰 전환 |
| 복잡한 폼 | 한 화면에 | 한 화면에 | 단계별(Wizard) |
| 드래그 앤 드롭 | 마우스 | 터치 롱프레스 | 미지원 |
| 알림 | 토스트 + 배지 | 토스트 + 배지 | 시스템 푸시 알림 |

## 모바일 전용 UX 설계

모바일에서는 데스크톱과 완전히 다른 네비게이션 패턴이 필요하다:

### 네비게이션

- **하단 탭 바**: 주요 4~5개 메뉴를 하단에 배치 (사이드바 대체)
- **스와이프 네비게이션**: 좌우 스와이프로 이전/다음 화면 전환
- **드로어 메뉴**: 전체 메뉴는 좌측 드로어에서 접근
- **뒤로 가기**: 브라우저/OS의 뒤로 가기 버튼과 앱 내 네비게이션 동기화

### 데이터 표시

- **테이블 → 카드**: 넓은 테이블은 카드형 리스트로 자동 전환
- **컬럼 우선순위**: 중요도가 낮은 컬럼을 자동으로 숨기고, 카드 탭/펼치기로 상세 확인
- **무한 스크롤**: 페이지네이션 대신 무한 스크롤 적용 (터치 친화적)

### 입력

- **큰 터치 타겟**: 최소 48×48px (Material Design 권장)
- **네이티브 입력**: 날짜/시간은 OS 네이티브 피커 사용
- **자동 저장**: 모바일에서는 뒤로 가기로 데이터를 잃기 쉬우므로 자동 저장 활성화

## PWA (Progressive Web App) 옵션

모바일 경험을 강화하기 위해 PWA를 적용할 수 있다:

| 기능 | 효과 |
|------|------|
| 홈 화면 추가 | 앱 아이콘으로 빠른 접근 |
| 오프라인 캐시 | Service Worker로 기본 Shell 캐싱, 오프라인 시 마지막 데이터 표시 |
| 푸시 알림 | 시스템 수준 알림으로 긴급 사항 전달 |
| 전체 화면 | 브라우저 UI 없이 앱처럼 표시 |

## 테스트 전략

전 디바이스 지원 시 테스트 매트릭스가 크게 확장된다:

| 카테고리 | 테스트 대상 |
|---------|-----------|
| 브라우저 | Chrome, Edge, Firefox, Safari (데스크톱 + 모바일) |
| 해상도 | 320px, 375px, 414px, 768px, 1024px, 1366px, 1920px, 2560px |
| OS | Windows, macOS, iOS, Android |
| 인터랙션 | 마우스, 터치, 키보드 |
| 네트워크 | 고속(WiFi), 저속(3G), 오프라인 |
| 방향 | 가로, 세로 (태블릿/모바일) |

## Mobile-First vs Desktop-First

| 접근 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **Mobile-First** | 모바일을 기본으로 설계, 큰 화면으로 확장 | 모바일 UX가 우수 | 데스크톱 복잡 기능이 어색할 수 있음 |
| **Desktop-First** | 데스크톱을 기본으로 설계, 작은 화면으로 축소 | 데스크톱 UX가 우수 | 모바일이 "축소판"처럼 느껴질 수 있음 |

산업용 포털은 데스크톱이 주력이므로 Desktop-First가 일반적이지만, 모바일 사용 비중이 높다면 Mobile-First를 고려한다.

## 제약 사항 및 고려사항

1. **비용 대비 효과**: 모바일 사용이 전체의 5% 미만이면, 투자 대비 효과가 낮을 수 있음
2. **기능 격차**: 모바일에서 사용할 수 없는 기능이 많아 사용자 혼란 가능
3. **성능**: 모바일 브라우저의 성능 한계로 대규모 데이터 처리가 어려움
4. **네트워크**: 모바일 환경은 네트워크가 불안정할 수 있어, 오프라인/저대역폭 대응 필요
5. **테스트 비용**: 테스트 매트릭스가 데스크톱 전용 대비 3~4배 증가
